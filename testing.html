Testing

Testing Strategy

We have developed a comprehensive testing strategy that encompasses various aspects of the system. Specifically, we plan to conduct unit testing for the backend, user acceptance testing to evaluate the user experience, and performance testing to assess the system's efficiency and speed.



Unit Tests

Unit testing was essential to ensure that our code functions as intended. In this particular case, the unittest module is being used to test the CadEditor class and its methods. This particular class has been chosen to test as it directly edits JSON making it simple to check if the unit test has passed or failed. It is important to note that only this class is being unit tested as our other functionality requires user inputs and is more suitable for user acceptance testing.


We developed nine unit tests to ensure that every function in cad.json was thoroughly covered. By creating a comprehensive suite of unit tests, we can be confident that each function in cad.json is functioning correctly and that any changes or updates made to the codebase will not cause unexpected issues or bugs. Every test case has passed indicating that our JSON has been edited as intended. 

class TestCadEditor(unittest.TestCase):
    def setUp(self):
        self.cad_editor = CadEditor()
        
    def test_change_tinkercad_align_phrase(self):
        # Change a phrase for the "align" command in Tinkercad
        self.cad_editor.change_phrase("tinkercad", "align", "align objects")
        updated_phrase = self.cad_editor.event_editor.get_data("speech_hotkey_tinkercad_align/args/phrase")
        self.assertEqual(updated_phrase, "align objects")

    def test_change_sketchup_circle_phrase(self):
        # Change a phrase for the "circle" command in SketchUp
        self.cad_editor.change_phrase("sketchup", "circle", "draw circle")
        updated_phrase = self.cad_editor.event_editor.get_data("speech_hotkey_sketchup_circle/args/phrase")
        self.assertEqual(updated_phrase, "draw circle")

    def test_change_cad_select_all_phrase(self):
        # Change a phrase for the "select_all" command in CAD universal program
        self.cad_editor.change_phrase("cad", "select_all", "select everything")
        updated_phrase = self.cad_editor.event_editor.get_data("speech_hotkey_cad_select_all/args/phrase")
        self.assertEqual(updated_phrase, "select everything")

    def test_update_cad_phrases_tinkercad(self):
        # Update multiple phrases for Tinkercad
        new_commands = {
            "align": "align new",
            "drop": "drop new",
            "flip": "flip new"
        }

        self.cad_editor.update_cad_phrases("tinkercad", new_commands)

        for command_name, phrase in new_commands.items():
            updated_phrase = self.cad_editor.event_editor.get_data(f"speech_hotkey_tinkercad_{command_name}/args/phrase")
            self.assertEqual(updated_phrase, phrase)

    def test_update_cad_phrases_sketchup(self):
        # Update multiple phrases for SketchUp
        new_commands = {
            "line": "draw line",
            "rectangle": "draw rectangle",
            "circle": "draw circle"
        }

        self.cad_editor.update_cad_phrases("sketchup", new_commands)

        for command_name, phrase in new_commands.items():
            updated_phrase = self.cad_editor.event_editor.get_data(f"speech_hotkey_sketchup_{command_name}/args/phrase")
            self.assertEqual(updated_phrase, phrase)

    def test_update_cad_phrases_cad(self):
        # Update multiple phrases for universal commands
        new_commands = {
            "select_all": "select all objects",
        }

        self.cad_editor.update_cad_phrases("cad", new_commands)

        for command_name, phrase in new_commands.items():
            updated_phrase = self.cad_editor.event_editor.get_data(f"speech_hotkey_cad_{command_name}/args/phrase")
            self.assertEqual(updated_phrase, phrase)

    def test_update_in_current_mode_with_current_program_phrase(self):
        # Update current mode with a phrase from the current program
        self.cad_editor.update_in_current_mode("speech_hotkey_sketchup_circle", "sketchup")
        self.assertIn("speech_hotkey_sketchup_circle", self.cad_editor.current_events)

    def test_update_in_current_mode_with_non_current_program_phrase(self):
        # Update current mode with a phrase not from the current program
        self.cad_editor.update_in_current_mode("speech_hotkey_tinkercad_align", "tinkercad")
        self.assertNotIn("speech_hotkey_tinkercad_align", self.cad_editor.current_events)

    def test_update_in_current_mode_with_universal_cad_command(self):
        # Update current mode with a universal CAD command
        self.cad_editor.update_in_current_mode("speech_hotkey_cad_select_all", "cad")
        self.assertIn("speech_hotkey_cad_select_all", self.cad_editor.current_events)

System Resource Utilisation Tests

To accurately monitor the CPU and RAM usage of MotionInput, we have developed a Python script that records the usage levels over a designated time period and generates a graphical representation of the data.

PC 1 Resource Utilisation
PC 2 Resource Utilisation

We evaluated the performance of MotionInput while being used in a CAD software and measured the CPU and memory usage of the application over a 5-minute period. Both PCs used in the test had 16GB of RAM, and we found that MotionInput used only 2% of the RAM on average, which is a positive result and indicates that it is not memory-heavy. This makes it more compatible with lower memory machines.


We observed that PC 1, which has a 10th generation Intel i7 CPU, had lower CPU usage than PC 2, which has a 4th generation Intel i5 processor. The CPU usage of MotionInput on PC 1 ranged between 6-10%, which is ideal as it minimizes the impact on the CAD software's performance. On PC 2, the CPU usage was between 15-20%, which is acceptable. However, we did notice some momentary spikes to 30-35%, which could be a cause for concern. Nonetheless, as they were only momentary, they should not be too much of an issue.


FPS Tests

A stable FPS is essential to prevent choppy visuals that can negatively impact user engagement. Thus we decided to test the performance over a period of time to ensure that it is suitable. We set up two PCs side by side, PC 1 and PC 2, provided them with identical visual input, and recorded the FPS every second.

PC 1 Resource Utilisation
PC 2 Resource Utilisation

The test results demonstrated that PC 1, featuring a 10th generation i7 processor, performed admirably, with peak FPS at 31 and a minimum of 24 FPS. Conversely, PC 2, equipped with a 4th generation i5 processor, reached highs of around 29 FPS but dipped as low as 16 FPS. These peaks and troughs correspond to periods of stillness and high movement in the visual input, respectively.


While PC 1 delivered optimal performance, PC 2 warrants further optimization. An FPS of 16 is barely acceptable, as it results in noticeably choppy visuals, albeit still functional. These findings suggest that systems with significantly outdated CPUs may struggle, although this represents a small market segment and is not our primary concern. Given the opportunity in the future, we could explore ways to reduce FPS drops during high movement, thereby enhancing compatibility with older devices.

User Acceptance Tests

User acceptance testing is crucial for assessing how well a software meets the expectations and requirements of our target audience.


Speech Phrase Test

The purpose of this test is to evaluate the capability of the custom dictionary functionality in recognizing words. The test is conducted with three participants who are asked to speak the words on a predetermined list. We check to see if a word has been recognised by seeing the partial_phrase detected by the vosk module. If the word is accurately recognised for all three users, a tick mark is recorded to indicate a successful recognition. This should allow us to see the performance boost from the custom dictionaries.


Our 3 participants all have varying voices and accents to ensure it can undestand words under different conditions. We also repeating the test with background noise, as that is a significant challenge that the exisitng MotionInput solution struggled with. The specific background noise that was used for this test was called Restaurant Ambience and was played on an external speaker to simulate people talking around the user.

Based on our user acceptance survey, we can conclude that the Hand Tracking Mode is the most effective and user-friendly option among all modes. It is easy to learn and operate, with test users rating it between 4 and 5 and completing tasks with relative ease. The primary challenge encountered was the small hitboxes for certain tasks, such as resizing, within the CAD software. Despite this issue, we are highly satisfied with the performance of Hand Tracking Mode.


Our Nose Tracking with Speech Commands Mode also performs reasonably well, with most functionality executed effortlessly, resulting in many 5 ratings. However, navigating the mouse with the nose proved challenging, leading to some 3 ratings. The small hitbox problem worsens this issue. Our current MotionInput version has difficulty interpreting leftward facial movements, which could be addressed in the future by transitioning to the MediaPipe facial recognition system, which offers smoother performance.


In contrast, the Nose Tracking with Facial Switches Mode received many 2 ratings, indicating poor performance. The nose movement is difficult to control, and the facial switches are not consistently triggered. Moreover, speech commands, which control much of the program's functionality, can inadvertently activate facial switch commands due to facial movement while speaking. This mode would also benefit from an improved facial recognition algorithm. Although this mode is functional, we only recommend it when no other options are available for the user.


Despite the challenges faced with the Facial Switches Mode, our users generally enjoyed the experience and provided positive feedback overall. These insights will enable us to refine and optimize our software as well.
